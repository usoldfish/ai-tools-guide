AI 新手 · 安装与工具地图

别再问“装了没”，先问：装的是谁，住在哪，谁在调用。

Claude、ChatGPT、Codex 这些名字，可能同时代表网页服务、桌面 App、终端命令和插件。名字一样，不等于同一个东西。

最重要的一句：安装方式 ≠ 安装位置；看到图标 ≠ 终端命令存在；终端能运行 ≠ 桌面 App 已安装。
支持系统分享；不支持时自动复制链接。
01

先记住一栋“AI 工具房子”

所有复杂名词都可以先放进这四个位置。

🏠
房间

桌面 App / GUI

能点开的完整应用。Mac 常住在 /Applications。

🚪
门牌

终端命令 / CLI

输入 claude 进入工具。门可能通向桌面 App，也可能通向独立程序。

🗄️
抽屉

配置与插件

账号、历史、Skill、MCP、设置，常在 ~/.claude 或 ~/.codex。

🛠️
工作台

项目目录

当前代码和项目专用配置。只服务这个项目，不该自动污染全电脑。

关键：一个“门牌”可以指向某个“房间”；删掉另一扇多余的门，不一定会删掉房间。
02

同一个品牌，可能有四种身份

GUI

桌面端

鼠标点击、窗口、菜单，适合聊天和日常使用。

≠
CLI

命令行

在终端里操作文件、代码、自动化任务。

PLUGIN

插件 / Skill / MCP

给某个宿主增加能力，通常不能单独运行。

≠
MODEL

模型 / 在线服务

云端大脑，不一定“安装”在你的电脑里。

你看到的东西它是什么典型入口能否独立运行最常见误会
Claude 桌面端GUI 应用启动台 / 应用程序可以以为必然自带同名 CLI
Claude CodeCLI 编程代理终端输入 claude可以以为它就是聊天桌面端
Claude 浏览器扩展浏览器插件Chrome 扩展栏不可以以为装进了整台电脑
MCP / Skill宿主能力模块Claude / Codex 内部通常不可以以为“安装成功”就会自动启用
03

“装在哪里”有五个层级

系统应用层整台 Mac 可见
/Applications/Claude.app
完整桌面 App 的常见住所
所有用户或当前用户
命令入口层PATH 会在这里找门
/usr/local/bin/claude
可能只是快捷入口或软链接，不一定是程序本体
终端全局可调用
用户全局层只属于当前账号
~/.npm-global　~/.local
终端安装器常把独立 CLI 放在这里
你的所有项目
宿主内部层由 App 自己管理
~/.codex/skills　~/.claude
插件、缓存、配置、会话；不是普通“全局命令”
只给该宿主使用
项目本地层跟着项目走
项目/node_modules　.mcp.json
版本更可控，团队更容易复现
只在当前项目
“全局”不是“装进系统核心”。它通常只表示“多个项目都能用”。用户全局、系统全局、宿主内部，三者仍然不同。
04

“谁安装的”与“装到哪”是两条线

动作你从桌面端点安装App 自己下载组件
→
结果 A捆绑在 App 内卸载 / 更新通常由桌面端负责
或
结果 B建立一个 CLI 门牌终端能调用，但本体仍可能属于桌面端
动作你在终端输入安装命令例如 npm、brew、官方脚本
→
结果 A用户全局 CLI多个项目可用，由原安装器更新
或
结果 B项目本地依赖只在当前目录有效，跟随项目版本
卸载原则：谁装的，优先让谁卸；谁管理更新，就让谁更新。不要直接删看不懂的文件夹。
05

重复安装为何会“明明更新了，却还是旧版”

症状 1

版本对不上

桌面端显示最新版，终端 --version 却还是旧版。

症状 2

卸载后还能运行

删了一个来源，但 PATH 找到了另一份同名命令。

症状 3

设置互相不认

两份程序读取不同的配置目录、登录状态或插件。

终端输入同一个命令时，电脑到底选了哪一份？
↓
先看门牌
command -v claude

显示正在执行的入口路径。

再看身份
claude --version

显示这扇门后面的版本。

↓
查安装器清单
npm list -g --depth=0

只回答 npm 全局装了什么。

查是否多门
type -a claude

可能列出多个同名入口。

不要看到两条路径就都删。先保留你想长期使用的来源，再用另一来源自己的卸载方式移除重复项，最后重新检查路径和版本。
06

AI 新手还会遇到的同类陷阱

Node / npm / npx

npx 跑过 ≠ 全局安装

npx 常是临时执行；项目依赖和 npm -g 也不是一回事。

Python / pip

pip 装了却 import 不到

常因 Python、pip、虚拟环境不是同一套。先确认解释器身份。

VS Code / Cursor

扩展只属于那个编辑器

在 Cursor 装的扩展，不保证 VS Code 也有；配置同步也可能分开。

MCP / Skill

文件存在 ≠ 已启用

还可能缺配置、权限、环境变量，或装在了另一个宿主里。

登录与 API Key

订阅账号 ≠ API 额度

聊天产品订阅、开发者 API 计费、CLI 登录，通常是不同通道。

多个账号

网页能用，CLI 却没权限

可能登录了不同账号、团队或组织，不是安装故障。

配置与历史

卸载 App ≠ 清空个人数据

应用、配置、缓存、会话常分开放；重装后旧设置可能回来。

沙盒与权限

看得到文件 ≠ 有权修改

系统权限、App 沙盒和项目授权，会限制可读写范围。

PATH

装好了却提示 command not found

程序可能存在，但终端的“找门路线”里没有它，或终端尚未刷新。

07

遇到问题时，按这个顺序问

我现在要用的是网页、GUI、CLI，还是插件？先确认身份，不要只说“Claude 坏了”。
我是从哪里启动它的？浏览器、启动台、终端、编辑器侧栏，入口决定排查方向。
真正执行的是哪个路径、哪个版本？CLI 用 command -v 与 --version；GUI 看“关于”。
它由谁安装、谁更新？桌面端、App Store、Homebrew、npm、官方脚本、插件市场不要混用。
问题是安装、权限、账号，还是网络？能启动但不能登录，通常不是“没装好”；能登录但读不到文件，通常是权限。
是否存在第二份同名入口？只有在证据确认后，才移除重复来源。

08

小白最值得养成的 8 条习惯

1. 先选一个主要来源

同一 CLI 尽量只保留官方推荐的一种安装来源。

2. 记录安装命令

知道怎么装，才知道以后怎么更新和卸载。

3. 安装后立刻验明身份

确认入口路径、版本和登录账号。

4. 项目依赖优先本地

需要固定版本时，别图省事全都装成全局。

5. 插件只从可信来源装

关注它会读什么文件、发什么网络请求、是否自动执行。

6. 不要混用 sudo 修一切

权限错误可能被暂时掩盖，却留下更难清理的归属问题。

7. 区分应用、配置与数据

卸载前先判断哪些需要备份，哪些可以清理。

8. 出错先分类，不要重装

身份 → 路径 → 版本 → 安装器 → 权限/账号/网络，逐层排查。

一句话毕业标准：你能说清“我在什么入口，用哪一份程序，读取哪里的配置，由谁负责更新”，就已经不会被大多数 AI 工具安装问题绕晕。
